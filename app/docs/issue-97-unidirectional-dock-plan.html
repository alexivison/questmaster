Implementation Plan · Phase 3 of the store→view migration

Unidirectional per-session dock view model

Retire the imperative push-to-views model and the record / restore bookkeeping that exists only to tame it. Make the store the single source of truth and the dock a pure function of it.

Issue  #97 Builds on  PR #93 Branch  claude/issue-97-fix-plan-k16cga ⚠ macOS only — AppKit can't compile on Linux/CI
The one-line thesis. Today the AppKit view owns dock state and the store is a parallel memory, so every user change must be recorded into the store and every session switch pushed back into the view — a bidirectional sync faked over one-way push. Fold all per-session dock state into one SessionViewState map, make renderSnapshot() a pure store→view projection, and the four pieces of machinery built to make that round-trip safe (isRestoring, the lag-idempotency argument, the silent-twin setter, the clobber guard) cease to have anything to suppress and are deleted, not patched.

00Decisions locked for this plan

Two questions shaped the plan's structure. Both are settled; the rest of the document assumes them.

PR #93 is open/unmerged and #97 retires the bookkeeping it adds. How do we sequence them?

Build on top of #93

#93 lands first as the contained, unit-tested Core patch it already is. #97 is the follow-up PR that removes the machinery. Matches the issue's "done in #93 instead" framing. This plan starts from the post-#93 tree.

Reconcile the two "active session" pointers (RuntimeStore.currentTerminalSessionID vs the store's activeSessionID)?

Include the reconciliation

One notion of "which session, and what it shows." The new view-model store derives its active session from RuntimeStore rather than carrying a second pointer that lags. Fully addresses the issue's third bullet.

01The problem, precisely

"Dock state" is spread across five owners at three layers. Because the view is the source of truth, the store is a shadow copy that must be kept in sync by hand in both directions.

dockVisible, focus region
DockViewAppKit · view
live contentMode + artifactRoute — the painted truth
RuntimeStoreCore · runtime
currentTerminalSessionID — the viewed session
SessionUIStateStoreCore · per-session
per-session dockVisible + DockContent (added by #93)
ArtifactDisplayStateCore · inside DockView
per-session selectionBySessionID + known paths

…record→ store

The two directions fight: a restore write would echo back through the recording sinks, so it must be suppressed (isRestoring); a record can fire mid-switch while the store's active id still lags, so it must be a no-op by construction (the lag invariant); and renderSnapshot fires from ~22 non-switch sites, so restore must early-out unless the session truly changed (the clobber guard).

The tell that this is architectural debt, not feature complexity: adding the next per-session facet (tracker visibility, dock width, scroll position) means repeating the record + restore + silent-setter triad across three files, each with the ordering/echo footgun. #93 paid the cost once and contained it; this is the moment — the second facet — to flip it.

02The target shape

One owner per session. The view receives its state each render and never stores it. User actions write the store directly; there is no separate "record" step because rendering can't write back.

After — unidirectional store → render

User action
store.setDockContent(.artifactViewer, for: active)
→
SessionViewStateStore
single per-session source of truth (mutates)
→
renderSnapshot()
compute desired state for active session
→
DockView.apply(state)
diff vs painted → apply. Owns nothing.

Renders become idempotent by construction: applying the same desired state twice paints the same pixels and writes nothing back. A session switch is just "the active id changed → the desired state changed → re-render." No suppression window, no lag invariant, no twin setter, no clobber guard.

The single per-session value type

Fold dockVisible, dockContent, and selectedArtifactID into one struct. This subsumes SessionUIState and absorbs ArtifactDisplayState.selectionBySessionID. The knownArtifactPathsBySessionID derivation-cache stays (it powers new-artifact auto-open detection) but is GC'd by the same prune.

// app/Sources/Core/Navigation/SessionViewState.swift  (rename/extend SessionUIState)
public struct SessionViewState: Equatable {
    public var dockVisible: Bool
    public var dockContent: DockContent        // .board / .artifactList / .artifactViewer
    public var selectedArtifactID: String?    // ← absorbed from ArtifactDisplayState

    public init(dockVisible: Bool = false,
                dockContent: DockContent = .board,
                selectedArtifactID: String? = nil) { … }

    public static let initial = SessionViewState()
}

The store: writers in, no record-back

Rename SessionUIStateStore → SessionViewStateStore. It keeps the id-keyed map and pruneSessions(keeping:), but its surface flips from "record live UI / restore into UI" to "mutate state for a session / read desired state for the active session." Crucially it derives the active session from RuntimeStore instead of holding a second pointer — that is the reconciliation.

// app/Sources/Core/Stores/SessionViewStateStore.swift
@Observable public final class SessionViewStateStore {
    private var statesBySessionID: [String: SessionViewState] = [:]

    // Reconciliation: there is ONE viewed-session source — RuntimeStore. The active id is
    // passed in at read/write time (or injected), never stored & advanced separately.
    public func state(for id: String?) -> SessionViewState {
        guard let k = cleanSessionID(id) else { return .initial }
        return statesBySessionID[k] ?? .initial
    }

    // The ONLY writers. Called directly from user-action handlers. No suppression — 
    // rendering never calls these, so nothing can echo.
    public func mutate(_ id: String?, _ body: (inout SessionViewState) -> Void) {
        guard let k = cleanSessionID(id) else { return }
        var s = statesBySessionID[k] ?? .initial
        body(&s)
        statesBySessionID[k] = s
    }

    public func setDockVisible(_ v: Bool, for id: String?)        { mutate(id) { $0.dockVisible = v } }
    public func setDockContent(_ c: DockContent, for id: String?) { mutate(id) { $0.dockContent = c } }
    public func setSelectedArtifact(_ a: String?, for id: String?) { mutate(id) { $0.selectedArtifactID = a } }

    public func pruneSessions(keeping liveIDs: Set<String>, active: String?) { … spare active … }
}

Note what's gone vs #93: no activeSessionID stored property, no isRestoring, no restoreIfActiveChanged, no record. The active id is always the caller's RuntimeStore.currentTerminalSessionID.

03Implementation steps

Ordered so the Core layer (testable on Linux) lands first and the AppKit rewrite (macOS-only) lands last against a stable Core surface. Steps 1–3 are CI-verifiable; steps 4–6 require a Mac.

1

Introduce SessionViewState + the writer-only store Core

new/rename SessionViewState.swift · SessionViewStateStore.swift  from SessionUIState.swift · SessionUIStateStore.swift

Extend the value type with selectedArtifactID. Replace the store's record/restore/suppression API with the mutate / state(for:) surface above. Keep pruneSessions but take the active id as a parameter (no internal pointer to spare). This compiles and tests on Linux — do it first and prove it green.

2

Move artifact selection out of ArtifactDisplayState Core

edit ArtifactDisplayState.swift

Delete selectionBySessionID and the selectedArtifactID computed property; selection now lives in SessionViewState. ArtifactDisplayState keeps only the derivation role: given a snapshot + the session's selected id, produce ArtifactDisplayUpdate (artifacts, display state, new-artifact intent). The new-artifact detection via knownArtifactPathsBySessionID stays. select / moveSelection / recoverSelection become pure functions of (artifacts, currentSelection) → newSelection that the caller writes back into the store — no internal per-session map.

This is the trickiest Core change: moveSelection and "recover the remembered artifact, else fall back to first" must keep working. Keep them as pure helpers returning the new id; the App layer calls store.setSelectedArtifact(newID, for: active).

3

Rewrite the Core tests to the new surface Tests

edit SessionUIStateTests.swift → SessionViewStateTests.swift · ArtifactCoreTests.swift · main.swift

The two tests that exist only to pin retired machinery get deleted, not migrated: restoreSuppressesRecordingDuringApply (there is no suppression) and preSwitchRecordHitsOldSessionThenRestoreLoadsNew (there is no lag to be idempotent over). Replace with tests for the new invariants: mutating session A never touches B; reading A's state after a B-detour returns A's exact value; prune spares the active id; the artifact selection round-trips through the store. Update QuestmasterLogicTests.main's call list.

4

Make DockView a receiver, not an owner AppKit · macOS

edit app/Sources/App/Quests/DockView.swift

Delete the stored contentMode and artifactRoute as sources of truth. Add one entry point — apply(_ desired: SessionViewState, snapshot:) — that sets pane visibility + route + renders from the passed state, diffing against what's painted. Remove the onModeChanged / onWidthModeChanged recording callbacks; user gestures inside the dock instead call back "intent" closures (e.g. onOpenArtifact(id), onShowList, onSelectBoard) that the App layer turns into store writes. Delete applyRestoredDockContent — its only reason to exist (a silent twin that doesn't echo) is gone; apply is the one writer of view fields.

Keep currentWidthMode as a pure derivation of the desired state (viewer/list/board), computed during apply — the App layer reads it to drive split width. It is output, not stored truth.

5

Collapse renderSnapshot to a one-way projection AppKit · macOS

edit app/Sources/App/AppDelegate.swift  (renderSnapshot, applyNavigationState, switch paths)

The new body, per render:

private func renderSnapshot() {
    let active = runtimeStore.currentTerminalSessionID        // the ONE viewed-session source
    let desired = sessionView.state(for: active)             // pure read, no restore/guard

    navigation.setDockVisible(desired.dockVisible)            // focus-preserving (kept from #93)
    dockView?.apply(desired, snapshot: runtimeStore.snapshot) // view paints from state, owns nothing
    splitView?.setDockWidthMode(dockView?.currentWidthMode ?? .standard, animated: false)

    applyNavigationState()
    let live = Set(runtimeStore.snapshot.tracker.repos.flatMap(\.sessions).map(\.id))
    sessionView.pruneSessions(keeping: live, active: active)
}

Delete from applyNavigationState the trailing sessionUIState.record { $0.dockVisible = … } line — visibility is now written by the user-action handlers (toggleDock, hideDock, restore-driven setDockVisible). Delete the onModeChanged record sink in configure (line ~317). Every render is now idempotent, so the "only restore on real change" guard is unnecessary and removed — re-rendering the same active session is a no-op against the painted state.

6

Route user actions to direct store writes AppKit · macOS

edit AppDelegate.swift · DockView.swift wiring

Wire the dock's intent closures to sessionView.setDockContent(…, for: active) / setSelectedArtifact(…, for: active), then call renderSnapshot(). Same for toggleDock → setDockVisible(…, for: active), and the new-artifact auto-open intent from setSnapshot → write selection + content then render. Confirm the focus-never-stolen behavior holds: visibility changes still go through navigation.setDockVisible (which #93 made focus-preserving). End state: one writer (user actions), one reader (renderSnapshot), one source (the store).

04What this deletes

Each retired item maps to a problem in §01. The point of the refactor is that these aren't suppressed or refactored — there is nothing left for them to do.

DELETE
isRestoring suppression windowNo echo to suppress: rendering reads the store, it never writes back. The flag + the guard !isRestoring in the writer disappear.
DELETE
The activeSessionID-lag idempotency argumentNo record fires during a half-finished switch because there is no record step and no second active-id pointer that lags. The invariant — and its can't-run-on-CI fragility — is gone.
DELETE
applyRestoredDockContent (the silent twin)DockView.apply(_:) is the single writer of contentMode/artifactRoute; there is no interactive-vs-silent split to keep in sync.
DELETE
The "re-restore only on real change" clobber guardRenders are idempotent by construction, so the early-out that protected an in-progress user change is unnecessary.
MERGE
One of the two parallel per-session mapsArtifactDisplayState.selectionBySessionID collapses into SessionViewState. knownArtifactPathsBySessionID stays (derivation cache) under the same prune.
MERGE
Two "active session" pointers → oneRuntimeStore.currentTerminalSessionID is the sole viewed-session source; the store's separate activeSessionID is removed.

05The acceptance proof: next facet = one field + one line

The issue requires demonstrating that adding a per-session facet under the new model is trivial. Document this in the PR with a real example, e.g. tracker visibility.

// 1 — one field on the value type:
public var trackerVisible: Bool = true

// 2 — one line in the render projection:
splitView?.trackerVisible = desired.trackerVisible

// (writes come for free from the existing toggle handler:
//  sessionView.mutate(active) { $0.trackerVisible.toggle() })

Contrast with today: the same facet needs a field, a record sink, a restore branch, a silent setter, and care around the echo/lag footguns — across three files. That delta is the justification for the refactor.

  • Smoke A→B→C round-trips: session A on the quest board, B on the artifact list, C in the viewer showing a specific artifact X. Switch among all three repeatedly — each restores its exact screen; C returns to X, not the first artifact.

  • New-artifact auto-open still fires for the viewed session only.
  • Session end (delete/exit current) closes the dock to defaults; no stale restore after delete.
  • Focus is never stolen on switch (visibility changes go through the focus-preserving setDockVisible).

07Risks & gotchas

HIGH
App-layer is compiler-unverified until a Mac runs it. Steps 4–6 touch AppDelegate / DockView, which can't build on Linux/CI. Sequence Core-first (1–3) so the surface the App layer targets is already green; treat the macOS build as the real gate. Do not push App-layer edits as "done" without a local swift build --package-path app.
HIGH
Dock width ordering. currentWidthMode derives from content + route. #93 fixed a bug where width was pushed before route was set. Keep width derivation/push after apply sets the route, now that route comes from the passed state rather than a reset.
MED
New-artifact detection must survive the selection move. knownArtifactPathsBySessionID stays in ArtifactDisplayState; the auto-open intent still keys off "viewed session + a genuinely new path." When auto-open fires, write both the selection and dockContent = .artifactViewer to the store, then render — don't rely on the view to remember.
MED
The freshly-attached-session prune race. #93's prune spares the active id because its tracker entry can lag a render. Preserve that: pruneSessions(keeping:active:) must union the active id even when it's absent from the live set.
MED
Observation correctness. @Observable on a class only fires on property assignment; mutating the dictionary in place is observed, but follow NavigationStore's copy-assign-back idiom if any consumer reads the store reactively. For now the only reader is renderSnapshot (still imperative), so this is latent — but keep the assignment discipline so a later SwiftUI reader works.
