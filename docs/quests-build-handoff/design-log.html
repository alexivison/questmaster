00 North Star

// the one sentence everything serves

what a quest actually is

The plan file you already make by hand for every ticket — your research, orientation, progress, next steps — turned into something structured: viewable, editable, tracked, and machine-readable so it also guides and gates the sessions you spawn.

Not a tracker. Not a layer over Linear. Linear holds the assignment someone filed; the quest holds your plan and what counts as done — a separate artifact you already keep (today as HTML files + an always-open index). The quest makes it real instead of hand-rolled. The strongest justification a personal tool can have: it automates something you already do.

01 Two Views of a Quest

// terminal for the index · browser for the plan

You left markdown behind because you wanted to see the plan — and the TUI mocks proved HTML wins. So a quest carries two representations:

terminal · summary
status · gates · sessions · next steps. The cockpit index — always open, scannable.
browser · the plan
the rich HTML body. One keystroke in the cockpit opens it in your browser.

The terminal never renders HTML — it just knows how to launch it. Structured metadata up top (terminal), rich plan body underneath (browser).

02 Two Stages

// the quest is the boundary between them

PLAN — create the quest full control
Interactive, tmux-hosted session(s). You talk with a master, feed it Linear links, read code together; the master sends & receives from workers live while you work. Jump between several at once. This is current Questmaster's strength — kept.
▼   the quest takes shape   ▼
EXECUTE — run the quest lightweight
Send a session on the quest and await. No live coordination needed — the inbox / messages-after-you-leave model is fine. You do manual checks, then mark done at merge.

Planning is quest creation — the front half of the lifecycle, not a separate activity. Before the quest exists you're in rich interactive planning (producing it); after, in lightweight execution (consuming it).

PLAN talk · links · read code · jump sessions→quest takes shape→DISPATCH send a session on it · await→VERIFY manual checks→DONE at PR merge

03 Sessions, the Layer & Comms

// tmux demoted: from message-carrier to session-host (not retired)

Sessions have a role (solo · master · worker) and run interactive (planning — the real native TUI) or headless (execution — supervisor-owned, survives close). The cockpit renders an always-on roster of all sessions across all repos.

The router owns messaging — addressing, logging, format. tmux is no longer the transport; it's the session host for the interactive planning stage (and gives you the jump-between-sessions). Delivery adapts to where a session lives:

planning pane→live pty-write|headless→supervisor inject|solo native drive→queue + statusline → drain on exit

tmux stays optional to run (execution is headless; planning degrades to cockpit-steer without it) — but it's where the planning experience earns its keep, so you keep it.

04 The Quest — File, Template & Gates

// one file · authored vs observed · conformance · definition of done

quest {
  goal string
  plan html body // the rich, browser-viewable plan
  gates[] Gate(auto|toggle) // definition of done
  context[] SourceRef // Linear / Slack / Notion
  worktree path // shared, single writer
  status derived
}

One fact, one home. Authored content — goal, gates, next-steps, context, the rich plan body — lives once in the quest file. Observed content — gate results, sessions, PR/CI — is written by the harness to a runtime record, never authored. Both views compose from these; nothing is stored twice, so drift is impossible by construction.

The file carries three layers:

  • injected
    Status strip — gate results, sessions, PR; rendered by the harness at view-time. Not authored.
  • canonical
    JSON head — the structured contract, shown as a visible code block: terminal & harness parse it, and you read it to audit against the prose.
  • authored
    Rich HTML body — the plan you and the planner write, diagrams and all. Browser renders it; the terminal links to it.

Conformance, two layers. Structure is machine-enforced: on every create/edit the harness validates the JSON head against a schema and refuses a malformed quest, feeding the error back to the planner — the same refuse-and-re-engage loop as gates. Meaning is human-audited: a schema can't tell whether the prose agrees with the head, so the visible JSON lets you catch a gate or step that's in one but not the other and tell the planner to reconcile.

Gates are declared in the quest, each typed auto (a runnable check — ci · pr · test · typecheck) or toggle (a box only you flip — ui · smoke). Checkability by construction: auto needs a real check, else it's a toggle. The win that isn't redundant with anything: Claude can watch CI/PR; Codex and Pi can't — so the gate loop gives any agent "iterate until green."

Enforcement: the agent can't self-certify; gates are measured externally against the artifact; failures inject back as the next turn, to budget, then escalate. Guarantee: no false "done."

05 Authoring & Adapters

// the plan is co-written; trackers feed it

The quest is drafted in the planning stage — you + a master + research workers, live, reading code and the linked context. Agents can also create & self-attach quests; trust scales with gate type (externally-measured & not agent-defined = free; agent-authored tests = your approval). All agent-created quests are visible & overridable.

Linear / Notion / Slack are context sources (read via MCP) plus a thin status write-back. The quest store is the source of truth for the plan, gates, and progress.

06 Build Path

// earn each stage · prove value before the risky parts

Stage 1 — the plan layer
  • Build quest = plan file (HTML body + structured head)
  • Build cockpit: index + roster + PR/CI status + open-in-browser
  • This is the original Projdash+QM vision. Replaces your manual HTML files. Least risky — no headless rebuild, no loop yet.
Stage 2 — gates + loop
  • Add gate catalog + runner
  • Add iterate-to-green loop (the Codex/Pi gap)
  • Layered onto the quests from Stage 1.
Stage 3 — autonomous
  • Add headless walk-away execution
  • Add agent-initiated quests · daemon · DAG
  • The big, commodity, risky part. Only if earned.
Stage 0 — the cheap test
  • One real task: write gates, run the loop by hand on current QM.
  • Kill criterion: if it doesn't beat "QM + AGENTS.md gates + CI + discipline," adopt/don't-build instead.

07 Decisions & Open

// the record + what's left

  • North
    Quest = your plan file, formalized. Two views (terminal summary + browser HTML). Replaces the manual HTML+index workflow.
  • Shape
    Two stages — plan (interactive, tmux-hosted, live) and execute (lightweight, inbox). The quest is the boundary.
  • tmux
    Demoted, not retired: session host for planning, not the message transport. Router owns addressing; delivery is mode-aware.
  • Data
    One file: visible JSON head + rich HTML body; authored vs observed split, nothing stored twice. Structure machine-validated on save (refuse-and-re-engage); meaning human-audited via the visible JSON.
  • Locked
    Gates typed auto/toggle, agent-immutable when live; enforcement = no self-certify + external measurement + re-engage loop; agent-agnostic iterate-to-green.
  • Locked
    Sessions are the substrate; quest is an optional hat. One shared worktree per quest, single writer; sequential attempts.
Open · into the build spec

Async-gate state machine (suspend on slow CI/PR) · escalation round-trip (notify→reply→resume) · the steer/live-view fidelity spike · build-vs-adopt (Symphony spec / Archon overlap).

Landscape · one line

The category ("harness engineering," PEV loops) crystallized and is crowded — Archon, Symphony, Composio all overlap. The architecture is commodity; your edge is that it's tailored to exactly how you work, terminal-first, plan-file-centered. A personal-tool edge, and a real one.
