Quest · ENG-142

Auth refresh loop

Refresh tokens without forcing users to re-login.

linear:ENG-142slack:#authnotion:RFC-9

01 Context & Research

The current flow uses a sliding-expiry refresh: each call extends the same refresh token's life. The bug is that once a token crosses its absolute max age, the next refresh 401s and the user is bounced to login mid-session. The #auth thread and RFC-9 both point at rotation as the fix.

Reading auth.go and token.go: the refresh path is in refreshToken(); the retry wrapper is retryWithBackoff(). The 401 surfaces from the wrapper, not the refresh.

02 Approach

Switch to rotating refresh tokens: every refresh issues a new pair and persists it, so there's no absolute-age cliff. On a 401, refresh once, then replay the original request rather than failing out.

request → 401 → refreshToken() → rotate + persist → replay original → 200

03 Steps

  • Rotate the refresh token on every use (not sliding expiry)
  • Retry the original request once after a successful refresh
  • Manual: confirm no re-login across a token expiry

05 Open questions

Should old refresh tokens be revoked immediately on rotation, or kept for a short grace window to survive races? Decide before the retry step.
